The main algorithmic limitations come from both the graph model itself and the specific metrics computed on top of it.

Dijkstra's algorithm is the foundation for almost everything in this project, since all-pairs shortest paths are computed by running it once from every node. That makes the approach relatively expensive: as the network grows, recomputing every shortest path becomes slow, especially because betweenness_centrality, global_efficiency, average_shortest_path_length, and the edge recommendation search all depend on those distances. It also assumes non-negative edge weights, so it would not work correctly for any model with negative costs or incentives.

average_shortest_path_length only averages over reachable pairs and skips unreachable ones. The limitation is that it can make a fragmented network look better than it really is, because disconnected pairs are excluded instead of heavily penalizing the score.

network_diameter takes the maximum finite shortest-path distance and ignores unreachable pairs. In a disconnected graph, the true diameter is often treated as infinite or undefined, but this implementation reports only the largest distance inside connected portions. So it understates how bad fragmentation is.

global_efficiency treats unreachable pairs as contributing 0, which is standard, but it still compresses a lot of network behavior into one scalar. A small efficiency improvement does not necessarily mean the change helps riders in practice, because the metric does not account for demand, congestion, transfer inconvenience, line capacity, or different passenger volumes across stations.

betweenness_centrality is useful for identifying bridge stations, but it has several limitations here. It only measures how often a node lies on shortest paths, so it assumes riders always choose shortest routes. It does not model rider demand, service frequency, crowding, reliability, or the fact that multiple nearly-equal routes may split traffic. In this project it is also unnormalized, so the raw scores are mainly useful for ranking stations within this graph, not for comparing against other graphs of different sizes.

The edge recommendation algorithm, find_best_new_connection, is a brute-force search over candidate non-edges. Its biggest limitation is computational cost: for each candidate edge, it copies the graph and recomputes global efficiency from scratch. That becomes expensive quickly. It is also simplified conceptually. It estimates new edge weight from geographic distance and observed seconds-per-km rather than real engineering or schedule data. It optimizes only for global efficiency, not cost, demand, construction feasibility, land use, or political constraints. It also filters candidates using heuristic distance and path thresholds, so the best edge is only best within the searched subset, not necessarily globally best.

There is also a modeling limitation in how the graph is built. The TTC network is represented as an undirected weighted graph where edge weights are average travel times. That loses several real-world details. Directionality is ignored, even though transit travel times can differ by direction. Repeated trips are averaged into one edge weight, so time-of-day variation disappears. Stations are aggregated by normalized name, which can merge distinct platforms or operationally different stops into one node. Transfers, wait times, delays, and service frequency are not explicitly modeled.

For libraries, the main added Python dependency is plotly. Plotly is great for interactive output, but it is only a visualization layer. It does not improve the correctness of the graph algorithms, and large graphs can become cluttered or slow to render because every station and edge becomes part of an interactive figure. It is also primarily descriptive: it helps users inspect results, but it does not help with optimization, simulation, or uncertainty analysis.

tkinter in the GUI is lightweight and built in, but it limits the app to a desktop-style interface and fairly basic widgets and canvas rendering. For dense transit networks, custom drawing on a canvas can get visually crowded, and Tkinter is not ideal for highly advanced interactive mapping compared with dedicated GIS or web frameworks.

The 3D map output uses MapLibre and OpenFreeMap through generated HTML rather than a Python geospatial library. Its limitation is that Python is mostly just packaging data into GeoJSON-like dictionaries and HTML; the heavy lifting happens in the browser, so the Python side has limited control over deeper geospatial analysis.
